## 原因

生产站浏览器自动化可以读取页面和 DOM，但在点击 `+ 添加参考` 时反复超时。命令行 API 又会被 Sites 登录层拦截。为了让后续每轮部署后可以稳定复测 CRUD，需要把生产 smoke test 从脆弱的浏览器点击链路中解耦出来。

## 影响

- 普通浏览器请求不带 e2e header 时保持原行为。
- 带错误 e2e token 的脚本请求会返回 `401`。
- 正确 token 由部署环境变量 `REF_FORGE_E2E_TOKEN` 提供，不写入仓库。
- 当前生产 Sites 访问策略仍会在请求到达 Worker 前拦截命令行请求，因此还需要 Sites 访问策略配合才能完成真实生产写入复测。

## 风险 / 遗留问题

- `REF_FORGE_E2E_TOKEN` 还没有配置到生产部署环境。
- Sites access policy 还没有为该复测路径放行 token-bearing 请求。
- 因此本轮实现的是可维护的复测基础设施，不等同于第五轮生产 CRUD 已通过。
